# HyperSolid Phase 3.2 — 幂等账本持久化（生产级 / expo-sqlite）实施计划（continuous-agent-loop）

> **驱动方式：** 本计划由 `continuous-agent-loop`（sequential + quality gates）逐单元推进。
> 每轮只完整交付**一个**未打勾单元：TDD → 实现 → 质量门 → 自检 → 打勾 → 停。
> **可重入**：每轮先读本文件，选第一个未打勾单元，不假设从零开始。
> 子技能建议：superpowers:test-driven-development（资金敏感，建议配 worktree + PR 评审）。

**目标：** 把 Phase 3 已建但**生产路径未接通**的 cloid 幂等账本，加固为**真能防重复/孤儿单**的持久化账本：先持久化 cloid 再签名 → 重试复用同一 cloid → 按 cloid 对账 → 启动恢复。**生产级标准**：按行 ACID 写、并发安全、崩溃一致、可审计、可观测。**绝不下真单。** 全程 tsc 零错 + jest 全绿（基线 326）。

## 已定技术选型（生产标准，禁止改动）

- 存储：**expo-sqlite**（按行 ACID/事务、带索引、可演进为统一本地库）。为可测，`SqliteIntentStore` 依赖注入极简 SQL 端口 **`SqlDb{ run/all/exec }`**，单测注入内存 fake（**不在 jest 跑 expo-sqlite 原生**）。
- at-rest：DB 文件设 **iOS Data Protection（`NSFileProtectionCompleteUntilFirstUnlock`）**为基线；SQLCipher 留作上线强化（需 EAS 构建，本阶段不引入）。
- 作用域：账本按 **`address × network(mainnet/testnet)`** 隔离（切钱包/网络不串单）。
- 保留：pending/submitted 留到对账完成；终态(filled/rejected/canceled) **保留 30–90 天 + 上限**，启动清理；schema **带版本号 + migration 钩子**。
- 重试：失败/回执不确定的单**显式「重试」按钮复用同一 cloid**；仅明确网络超时可自动重试（同 cloid）。
- 恢复：启动扫 pending/submitted → `info.orderStatus(by cloid)` 查 HL → 对账；孤儿（HL 查无）标「未确认」让用户复核（不擅自当成交/失败）。
- 可观测：账本关键事件（持久化/对账/孤儿/重试）打 **Sentry breadcrumb**（栈内已有 Sentry）。
- 落点：**长生命周期单例** ExchangeService + 持久 IntentLedger（zustand store 或模块单例，沿用 walletStore/envStore 范式）；`IntentLedger` 同步内核**不改**，仅在其背后加「内存缓存 + 异步写穿」持久层。

## 不可触碰 / 范围边界

- **生产标准**：选型/默认按「已定技术选型」执行，不退回 MVP 捷径。
- **绝不下真单**：所有测试注入式 mock（含 SqlDb fake / info fake）；CI/测试默认 testnet；禁止对主网或真实资金下单/撤单、禁止需真机签名的人工步骤。
- **不改 Phase 2 安全层**（biometricGate/sessionController/authStore/deviceIntegrity/secureKeyStore）。
- **不改 Phase 3 编码核心**（order/buildOrder/cancel/cloid）与 **IntentLedger 同步内核**——只在其 IntentStore seam 后加持久层；可像 Phase 3 注入式扩展 ExchangeService。

- 基线：当前 326 单测通过、tsc 零错。任何一轮结束都不得让其下降。

#### 生产清单自检（逐项核对）

- **ACID 写**：`SqliteIntentStore.set/delete` 为单行 `UPSERT(ON CONFLICT DO UPDATE)`/`DELETE`，每条原子（expo-sqlite runSync 同步、单语句原子）。✓

- 2026-06-22（单元 1）：写穿用**同步**而非「异步写穿」。理由：expo-sqlite 提供同步 API（`openDatabaseSync` → runSync/getAllSync/execSync），同步写穿对账本是**更稳健的生产选择**（每条 UPSERT/DELETE 原子、崩溃一致、无 in-flight 异步写在崩溃时丢失）；单条 row 写 sub-ms，不阻塞 UI。SqlDb 端口因此定义为同步。
